# Pi routing uses the global models config

Pi Coding Agent routing is integrated by maintaining Pi's global `~/.pi/agent/models.json` model configuration rather than by building a Pi extension or managing Pi sessions. XXSwitch uses Pi's documented global path and does not introduce a custom path setting for the first integration. XXSwitch registers its local router as an `openai-responses` provider because the router's most complete Codex-facing surface is `/v1/responses`, with Chat Completions handled as an upstream compatibility detail. The first Pi integration only exposes Codex/OpenAI-compatible Route Models and does not generate a separate Pi provider for Claude Providers. The Pi model list is generated from XXSwitch's current route model list so Pi sees the same models that the router can actually route; generated Pi model IDs remain the original route model names so requests reach the router without another mapping layer. Generated Pi model entries include the Route Model context window so Pi estimates usable context correctly, mark reasoning support, and explicitly map Pi thinking levels to XXSwitch's supported Route Model reasoning levels including `xhigh`; they avoid speculative cost metadata. The router token is written as Pi's provider `apiKey` while being treated as a redacted secret in XXSwitch surfaces. Enabling Pi is treated like enabling any other Agent Client and automatically enables the local router, but applying Pi routing is rejected when no Route Models can be generated because Pi requires an explicit model list. When Pi is enabled, XXSwitch refreshes its Pi entries after saved Provider or router configuration changes that affect routing, but runtime health events do not rewrite Pi's global config. XXSwitch only owns its own Pi provider and generated model entries, preserving unrelated user-defined Pi configuration and restoring any pre-existing XXSwitch entries when disabled. Pi appears as another Agent Client on the existing Route page rather than getting a Pi-specific management page. This matches XXSwitch's Agent Client pattern of pointing local coding agents at the router, keeps the first Pi integration limited to model routing, and avoids taking ownership of Pi-specific agent behavior.
